# wrangler integration tests

Verifies that `rosie-skills` keeps working as the library
[wrangler](https://github.com/cloudflare/workers-sdk/tree/main/packages/wrangler)
consumes. Wrangler uses rosie as an **external (unbundled)** dependency: on
`wrangler setup --install-skills` it dynamically imports `rosie-skills` and
calls `rosie.agents()` then `rosie.install("cloudflare/skills", { global: true,
agent, lockfile: false })`. These tests guard that exact seam.

## Levels

- **Level 1 — contract** (`contract.mjs`): hermetic. Imports the freshly built
  package by **bare specifier** (validating the `exports` map / ESM resolution,
  not a deep path into `dist/`) and replicates wrangler's glue code. No wrangler
  install, fully offline.
- **Level 2 — e2e**: builds the real `wrangler` from the `workers-sdk` source
  with a pnpm override pinning `rosie-skills` to this checkout's build, so OUR
  rosie is bundled into `wrangler-dist/cli.js`. Then runs the real `wrangler
  setup --install-skills` binary and asserts the skills land on disk. Needs git
  + npm-registry access (clone workers-sdk, pnpm install its tree).

  Why a source build: wrangler no longer keeps `rosie-skills` as a runtime
  dependency. It bundles rosie into `wrangler-dist/cli.js` with esbuild at its
  own build time (rosie is a `devDependency`), so a published wrangler carries a
  frozen rosie snapshot and an `overrides` against an installed wrangler can't
  reach it. See `design/wrangler-integration-testing.md`.

Both point rosie at the regression suite's mock GitHub server
(`tests/regression/lib/mock_server.py`) via `ROSIE_GITHUB_BASE_URL`, serving a
`cloudflare/skills` fixture under `tests/regression/fixtures/sources/cloudflare/`.
No real github.com traffic.

## Running

```sh
./run.sh                        # both levels
./run.sh --no-e2e               # contract only (fully offline)
./run.sh --workers-sdk-ref X    # clone ref/branch/tag (default: main)
./run.sh --port 8788            # mock server port
./run.sh --keep-tmp             # keep scratch dirs for inspection
```

Env equivalents: `SKIP_E2E=1`, `WORKERS_SDK_REF=X`, `PORT=N`, `KEEP_TMP=1`.

## Notes

- The skills install runs in wrangler's command wrapper, *before* the `setup`
  body, so the e2e level asserts on installed files and tolerates `setup`'s own
  exit code (autoconfig fails in an empty project, which is expected).
- Level 2 confirms it bundled *this* build by grepping the esbuild metafile
  (`wrangler-dist/metafile-cjs.json`) for a `rosie-skills@file` input, i.e. the
  pnpm `file:` override rather than a registry copy.
- The wrangler build's DTS (type-declaration) step can fail on an unrelated
  undici `FormData` type mismatch. The CJS bundle still builds, so Level 2
  tolerates the build exit code and asserts on `wrangler-dist/cli.js` instead.
- Level 2 is a full monorepo build: it clones `workers-sdk`, runs `pnpm install`
  on wrangler's tree (~1 GB), and `turbo build`s wrangler plus its workspace
  deps. Browser downloads are disabled via `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD` /
  `PUPPETEER_SKIP_DOWNLOAD`. Use `--no-e2e` to skip it offline.
- `cloudflare/skills` tarball fixtures are generated by
  `tests/regression/fixtures/build.sh` (auto-run by `run.sh` when missing).
